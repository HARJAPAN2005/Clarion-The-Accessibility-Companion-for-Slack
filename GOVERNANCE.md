# Governance

Clarion is an open-source project that welcomes community participation.
This document describes how the project is governed and how decisions are made.

---

## Project Leadership

Clarion is currently maintained by its founding contributors. The maintainer
list is recorded in [CODEOWNERS](CODEOWNERS).

Maintainers are responsible for:
- Reviewing and merging pull requests
- Triaging issues and discussions
- Releasing new versions
- Upholding the [Code of Conduct](CODE_OF_CONDUCT.md)
- Setting the technical direction of the project

---

## Decision Making

### Day-to-day decisions

Routine decisions (bug fix approaches, minor feature scope, documentation
improvements) are made by the reviewing maintainer. Consensus among active
maintainers is preferred but not required for small changes.

### Significant decisions

Changes that affect the public API, architecture, release schedule, or project
direction require a discussion open for at least **5 business days** before a
decision is made. Open a GitHub Discussion in the **Ideas** category to start
one.

### RFC process

Major new features or changes to core behaviour follow an informal RFC (Request
for Comment) process:

1. Open a GitHub Discussion with the `RFC` label.
2. Describe the problem, the proposed solution, alternatives considered, and
   any open questions.
3. Allow at least **2 weeks** for community input.
4. A maintainer summarises the discussion and records the decision in the
   Discussion thread.
5. Implementation can begin once the RFC is marked **Accepted**.

---

## Becoming a Maintainer

Regular contributors who demonstrate consistent, high-quality contributions
and share the project's values may be invited to become maintainers.

The criteria are:
- Meaningful contributions over at least 3 months
- Understanding of the codebase and project direction
- Constructive and respectful engagement with the community
- Agreement to uphold the Code of Conduct as a maintainer

An existing maintainer nominates a new maintainer via a private discussion.
Appointment requires consensus among existing maintainers.

---

## Conflict Resolution

Disagreements within the community should be resolved through respectful
discussion in GitHub Issues or Discussions. If a consensus cannot be reached,
the maintainers make the final decision.

Violations of the Code of Conduct are handled separately from technical
disagreements. See [CODE_OF_CONDUCT.md](CODE_OF_CONDUCT.md) for the
enforcement process.

---

## Changes to This Document

Changes to this governance document follow the same RFC process as significant
technical decisions.
